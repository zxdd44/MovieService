# Movie Service (Spring Boot REST API)

Данный сервис был создан и предназначен для управления информацией о фильмах.
Проект обновлен для работы с реляционной базой данных и поддерживает сложные связи между сущностями.

## Составляющие программы

1. **Spring Boot приложение**: Реализовано на стеке Spring Web и Spring Data JPA.
2. **Расширенная область (Domain)**: Включает 5 связанных сущностей — `Movie, Director, Genre, Studio, Review`.
3. **Глобальная обработка ошибок**: Настроена через `@ControllerAdvice`, что обеспечивает единый формат ответа об ошибке для всех эндпоинтов.
4. **Валидация данных**: Входные DTO проверяются с помощью аннотаций группы `@Valid`.
5. **Логирование**: Настроено через **Logback** с разделением по уровням и ротацией логов.
6. **Аспектно-ориентированное программирование (AOP)**: Реализован аспект для автоматического логирования времени выполнения методов сервиса.
7. **Документация Swagger/OpenAPI**: Подключена интерактивная документация для описания всех эндпоинтов и моделей данных.
8. **Unit-тестирование**: Бизнес-логика покрыта тестами с использованием **JUnit 5** и **Mockito**.

## Эндпоинты (CRUD)

### 1. Базовые операции
* 'POST /api/movies' — **создание нового фильма**.
* 'GET /api/movies/{id}' — **получение по ID**.
* 'PUT /api/movies/{id}' — **обновление данных**.
* 'DELETE /api/movies/{id}' — **удаление**.

### 2. Поиск и фильтрация
* `GET /api/movies/search` — Сложный поиск с поддержкой пагинации (`Pageable`), фильтрацией по жанру (через Native Query и JPQL) или режиссеру.
* Реализован **in-memory индекс** на основе `HashMap<K, V>` для кэширования результатов поиска по составному ключу.

### 3. Массовые (Bulk) операции
* `POST /api/movies/bulk` — Массовое создание фильмов из переданного списка объектов.

## Структура данных (Status Codes)
В API статусы фильмов передаются в виде целых чисел:
* `0` — NOT_WATCHED (Не просмотрено)
* `1` — WATCHED (Просмотрено)
* `2` — DEFERRED (Отложено)
* `3` — ABANDONED (Брошено)

---

## Модель данных (Entity Relationship)
В проекте реализовано 5 основных сущностей:
* **Movie**: Основная сущность
* **Director**: Режиссер (Связь OneToMany с Movie).
* **Genre**: Жанр (Связь ManyToMany с Movie).
* **Studio**: Пользователь сервиса.
* **Review**: Отзыв пользователя (Связь ManyToOne к Movie).

## Управление связями (Fetch & Cascade)
* 'FetchType.LAZY': Используется для всех коллекций (например, в Director.movies), чтобы избежать избыточной нагрузки на БД.
* 'CascadeType.ALL': Настроен для связи Movie -> Review. При удалении фильма автоматически удаляются все связанные с ним отзывы.

## Решение проблемы N+1
Для оптимизации производительности при получении списка фильмов с их авторами реализовано решение
@EntityGraph: Декларативное описание графа зависимостей в репозитории.

## Транзакционность и целостность
* **Без @Transactional**: При ошибке во время сохранения отзыва сам фильм может остаться в базе (частичное сохранение), что нарушает логику.
* **С @Transactional**: Любая ошибка вызывает полный Rollback, гарантируя, что база данных останется в консистентном состоянии.

## Сложные GET-запросы
* `http://localhost:8080/api/movies/search?genre=Thriller&page=0&size=5`

## Использование Stream API и Optional
* В сервисном слое активно используется **Stream API** для маппинга коллекций и фильтрации данных.
* Применение **Optional** позволяет безопасно обрабатывать потенциальные `null` при поиске сущностей в БД.
